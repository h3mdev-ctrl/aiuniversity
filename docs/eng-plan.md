## Locked architecture (Andrew's call, 2026-07-03)

**Data + deterministic matcher + thin agent wrapper.** Three parts, clean separation:

```
  pack.yaml   (DATA — inert; the recipe; human-readable BEFORE you run it)
      │  name, version, steps[], modules[]
      │  each step: instruction · check{type,cmd,expect} · fixture? · fix? · on_fail
      ▼
  verify.py   (CODE — the ONLY thing that decides PASS/FAIL; ~200 lines)
      │  validate whole pack first → for each step: run check → match → verdict
      │  modules: load referenced pack.yaml, run its steps inline (id-prefixed)
      ▲                    │ PASS  |  FAIL(step_id, actual_output)
      │ called in verify   ▼
  SKILL.md    (AGENT — UX only, never renders a verdict)
         teach     → narrate each step, then verify
         verify    → call verify.py, report result
         remediate → apply step.fix (runnable), re-verify;
                     still red → STOP "human needed at step N"  (escape hatch)
```

Why this shape (the load-bearing reasons):
- **Determinism lives in code.** Success criterion "same pack + same install → same result"
  is only guaranteeable if a program judges PASS/FAIL, never an agent interpreting. The
  agent narrates and remediates; it never decides.
- **The recipe is auditable before it runs.** The yaml is inert data. The only executable
  strings are `check.cmd` and `fix` — both visible in the file. This IS the security
  section's "review the runbook before you run it" gate.
- **modules: recursion is trivial** — load the referenced pack's yaml, run its steps inline
  with a prefixed id. One level of nesting for v1 (modules don't include modules — YAGNI).

Rejected alternatives: all-agent skill (non-deterministic verdict, nothing to audit);
SKILL.md-only with no yaml (checks not machine-runnable, no clean module primitive).

## The step-granularity rule (Design Open Question #1 — resolved)

**One step = one externally-checkable state change: the smallest action a single check can
prove PASS or FAIL.** If you can't write one check for it, split it. If two actions share
one check, merge them. This guarantees "which step broke?" is always answerable — the whole
diagnostic promise.

## pack.yaml — v1 schema (glossed)

("schema" = the fixed list of fields a pack file is allowed to have.)

```yaml
name: gbrain-windows
version: 0.1.0                 # versioning from day one → packs can be "kept up to date"
steps:
  - id: install
    instruction: "..."         # verbatim action; may be per-OS (windows/macos/linux)
    check:
      type: command_succeeds   # one of the 5 v1 types (below)
      cmd: "gbrain --version"
      expect: "gbrain"         # meaning depends on type
    fix: "gbrain init --pglite"  # RUNNABLE command remediate applies on a red check
                                 # (mechanical steps only; a human/live step omits it)
    on_fail: "..."             # human 'why + what', shown when it STOPS (never executed)
    teach: "lessons/install.md"  # optional narration file for teach mode
    fixture: null              # only for correctness checks (recorded expected answer)
modules:                       # a step that runs another pack's runbook here
  - slot: layer-4-capabilities
    ref: gbrain-windows        # version-pinnable later; unpinned = latest for v1
```

### The 5 v1 check types (YAGNI-trimmed from the design's 7)

| type | proves | used by |
|---|---|---|
| `command_succeeds` | a command ran without error | most install/config steps |
| `contains` | output includes an expected string | doctor HEALTHY, version banners |
| `exact` | output equals an expected string exactly | tight config assertions |
| `regex` | output matches a pattern ("regex" = text pattern) | flexible version/format checks |
| `endpoint_reachable` | a service answers | MCP-server-is-live (gbrain tools respond) |

**Deferred to correctness-exam packs** (nsw-site-planning, later): `number_within(epsilon)`
and `json_subset`. Not needed by foundation + gbrain, so not built yet.

### Behavioural checks (the design's core rule, made concrete)

Rule: every layer's exam must include ≥1 behavioural check — proof the capability actually
CHANGES the outcome, not just that it's installed ("presence"). Implementation:

- A `behavioural` check runs `claude -p "<probe>"` in a fresh, headless (no-UI) session and
  `contains`-matches an expected marker in the answer.
  - gbrain: probe = "answer something only gbrain knows" → confirm the answer surfaced.
  - memory: probe = a cold-recall question in a fresh session → confirm the expected memory
    slug appears.
- **Honest limit:** the fully rigorous "…and it CHANGED the answer vs. a no-memory baseline"
  needs a two-run diff. That belongs to the phase-2 proctor. v1 states this per layer rather
  than pretending a one-shot check proves it. (Layer 1's behavioural recall is "verified by
  probe" in v1; "verified by diff" when the proctor ships.)

## The escape hatch — state machine (spec it tight)

```
  run check ──green──> next step
     │ red
     ▼
  apply on_fail (max ONE attempt)
     │
  re-run check ──green──> next step
     │ still red
     ▼
  STOP → surface "human needed at step N" + the actual output
         (never loop, never let the agent improvise past a red gate)
```

A loop here reintroduces the exact drift the product exists to kill. One remediation
attempt, then stop. Non-negotiable.

## Foundation umbrella — the 6 layers as top-level step groups

From the reference-machine audit (design Appendix). Each row = a step group; layer 4
declares the module slot.

| Layer | v1 check | behavioural check |
|---|---|---|
| 0 Identity | identity files present + `contains` name | "who am I / how talk to me?" probe → specific answer |
| 1 Memory | `memory_doctor` HEALTHY (`contains`) | cold-recall probe (verified by probe in v1) |
| 2 Constitution+routing | CLAUDE.md states invariants + RESOLVER present | — (structural is enough here) |
| 3 Guardrails | hooks present | forbidden action (read a secret) is BLOCKED — behavioural by nature |
| 4 Capabilities | **module slot** → gbrain pack runs here | each module's activation exam passes |
| 5 Durability | backup ran recently (`command_succeeds`) | decision log actually has entries (not just exists) |

## Test plan

Heaviest coverage on the matcher, because if the checker is wrong, every pack lies.

```
  match(type, output, expected)                ← unit tests, the core
    5 types × (1 pass case + 1 fail case)
         │
  verify.py on a FAKE pack (outputs faked via fixtures = recorded answers)
    ├─ all green            → PASS
    ├─ step 3 red           → reports step 3 (not 2, not 4)
    ├─ red→fix→green        → remediate PASS
    └─ red→fix→still red    → STOP "human at step 3"     [escape-hatch test, mandatory]
         │
  modules: foundation pulls in gbrain pack
    └─ failure inside module → id = "gbrain-windows/step-4"
         │
  bad-recipe rejection
    └─ missing check.type / unknown operator → refused up front, clear message
         │
  GOLDEN PATH (manual now, CI later)
    └─ real foundation + gbrain on the reference machine → PASS
```

- **T1 (mandatory):** escape hatch — fix fails → STOP with correct step number, no loop.
- **T2:** behavioural checks tested via a recorded probe answer (real live probe only on the
  golden path); keeps ~95% of logic repeatably testable while being honest about the last mile.

## Build order

1. `verify.py` matcher + the 5 check types (unit-tested first — it's the core).
2. `verify.py` runner loop: validate-whole-pack-first, run steps, escape-hatch state machine.
3. `pack.yaml` schema + validation (reject bad recipes up front).
4. `modules:` recursion (one level).
5. Thin `SKILL.md`: teach / verify / remediate wrapper.
6. **[HARD CHECKPOINT — same week]** Author the real gbrain-on-Windows pack from footgun
   notes; thread it through the foundation umbrella end-to-end. If the schema or composition
   doesn't fit a real module, fix it NOW.
7. Golden-path run on the reference machine → PASS.
8. Hand foundation + gbrain to a friend. Zero-help setup = wedge proven.

Deferred (CEO plan): authoring-by-demonstration, rot-CI + full CI/CD, pack signing +
sandboxing (HARD GATE before any open marketplace), Obsidian module, correctness-exam packs,
the fuzzy-pack proctor.

## Interaction design (the newcomer's felt experience)

The product has almost no visual surface — it's a command plus text files — so "design"
here is not pixels. It is the words and moments a nervous newcomer meets while being taught.
Three rules, written now so the first pack is authored WITH them, not patched later. These
are the felt experience of teach → verify → remediate; get them wrong and a technically
correct pack still feels like the days-of-pain it was meant to replace.

### Rule 1 — Teach-mode voice: a patient mentor, not a manual

- One step on screen at a time. Never a wall of steps.
- Say WHAT to do and WHY it matters, in that order, in plain English. Gloss any technical
  word the first time ("MCP — the wiring that lets Claude call an outside tool").
- Speak to a smart person who is new, not to an engineer. No unexplained acronyms.
- End each step by naming what the recipient should now SEE ("you should see a version
  number like `gbrain 0.42` — that means it's really installed"), so success is felt, not
  assumed.

### Rule 2 — The failure message: the highest-leverage words in the product

When a check goes red, the message is the difference between "I've got this" and "I'm lost."
Fixed shape, every time — never a raw error dump:

```
  ✗ Step 4 didn't pass yet: "gbrain's tools aren't answering."
     What this means:  gbrain is installed but Claude can't reach it — the wiring
                       (the MCP registration) is missing.
     Do exactly this:  <the on_fail command / edit, verbatim>
     Then I'll re-check automatically.
```

Four fixed parts: (1) which step + a plain-English name for what failed, (2) what it means in
one sentence, (3) the exact prescribed fix — never "figure it out," (4) reassurance that the
re-check is automatic. The `fix` field supplies the runnable command for part 3 (remediate
applies it); the `on_fail` prose explains it for the human message; the runner frames parts
1, 2, 4. Authoring rule: a mechanical step carries a runnable `fix:` (install/wire/link/create);
a genuinely-human step (an interview, "add to your PATH", a live probe) omits `fix` and stops
with `on_fail` prose — never shell-run English at the machine.

### Rule 3 — The stop-moment: a teacher asking for help, not a crash

The one time the pack gives up (fix applied once, still red) must read like a mentor who
cares, not a stack trace:

```
  ⏸ Let's get a hand here.
     Step 4 ("connect gbrain's tools") is still not passing after the fix, so I'm stopping
     rather than guessing — guessing is exactly what breaks these setups.
     What worked: steps 1–3 ✓.   Where we're stuck: step 4.
     Send this to Andrew (or your pack author): the step name above + the last output below.
     <actual output>
```

Never loop, never improvise past this point (matches the escape-hatch state machine above).
The tone target: the recipient feels *guided to the exit*, not *dropped*. This moment is rare
by design, so it's worth making it kind.

### How these get verified (design has a check too)

Consistent with the product's own philosophy — don't assume, confirm by watching. The real
test of all three rules is the golden-path friend hand-off (build step 8): if the friend gets
stuck and the failure message alone unblocks him without messaging Andrew, the interaction
design passed. If he has to ask what a message means, that message is the next fix. The
newcomer's confusion is the scoreboard.
